SmartCMS — Architecture & Flow

How the system is wired today: the layers, the environments, and the end to end flows from content creation through AI publish to the live WordPress page. SmartCMS is an editorial workflow layer on top of the client's WordPress, letting teams draft, collaboratively edit, route through multi role approval, preview on the live theme, and publish AI assisted content.

1 The six layers

SmartCMS is not one application — it is a set of cooperating services. The frontend orchestrates everything through n8n, which is the integration hub between the database, WordPress, the collaboration server, and the AI.

The frontend never talks to WordPress, Supabase, or the AI directly. n8n is the single integration boundary — which is why every capability is a workflow, and why the database is the source of truth for the editorial state while WordPress is the source of truth for published content.

4 The core flows

A · Content creation & collaborative editing

Editor opens a draft → TipTap editor ↔ Hocuspocus (live multi-user editing, Y.js) → content saved as a publication_request in Supabase (via n8n)

B · Approval workflow (multi-role governance)

Editor submits → n8n builds a workflow_instance + ordered steps from the chosen template → each step routes to its approver role (Clinical, Supervisor, Publishing Manager…) → approver reviews on the Comparison page: Approve & Post → advance the chain; on final step → PUBLISH Reject / Rework → back to the editor queue + status driven by getApprovalsByRole (instance + step state)

C · AI Publish (what actually goes live)

final approval → BPRA (BlogPublishRequestAction, n8n) → convertHtmlToAcf — editor HTML → ACF block JSON (LLM chain) → acf_blocks_api_manager writes post_content as ACF block markup → WordPress theme renders the blocks → LIVE PAGE

A published post is stored as a single ACF block (acf/blog-section-builder) whose JSON holds the article; the theme re-renders it fresh on every request.

D · WordPress Preview (reviewer sees the live theme)

Reviewer clicks WordPress Preview → n8n stage bridge stages the content + mints a signed token → smartcms-preview companion plugin renders it on the LIVE theme, reviewer-only (token→cookie) → shown in an iframe — no WordPress login needed

E · Go Live (one-click direct publish)

Reviewer clicks Go Live in the preview → goLivePreview webhook → plugin promotes the previewed content to the live post → the request is resolved (status → Published, workflow closed, audit logged)
Two publish paths, by design. Approve & Post = governed, step-by-step approval chain. Go Live = immediate, single-click publish of preview-confirmed content. Complementary, not duplicates.

5 The publish format — and why preview fidelity is hard

This is the single most important architectural fact. The editor edits rendered HTML, but WordPress stores and publishes ACF blocks. Publishing is a round trip:

stored ACF blocks ──theme renders──▶ HTML (the editor loads & edits this) │ edits ▼ convertHtmlToAcf (LLM) ──▶ ACF blocks ──theme re-renders──▶ LIVE PAGE

The editor holds the output of the blocks; publishing re-derives the source (blocks) and the theme renders them again. That is why a preview that renders the editor's raw HTML cannot perfectly match the published page, and why true 100% preview fidelity is an active design effort (render the real blocks via the theme, converting once and publishing that exact payload).

Current state: the preview renders the editor HTML directly (fast, reliable) — faithful for simple content, but not pixel-identical to the block-rendered published page for richly structured posts. The path to true 100% is documented separately.

6 Data model (the editorial engine)

The approval workflow is a small state engine in Supabase:

TableRole
publication_requestsOne per publish request — the edited content (final_html_content), the source post, status (Pending/Approved/Rework/Rejected/Published), reviewer, title.
workflow_instancesThe live approval run for a request — overall status (in_progress/approved/…).
workflow_instance_stepsThe ordered steps of an instance — each a role-gated approval gate.
workflow_instance_step_assigneesWho can action each step.
ai_audit_sessionsThe AI generation/edit session behind a request.
profiles · roles · permissionsUsers, their roles, and the component/endpoint permissions that gate the UI.

Two systems of record: Supabase owns the editorial/approval state; WordPress owns the published content. A publish is the moment the two are reconciled.

7 Key technical notes

n8n is the only integration boundary

Every capability is a workflow. The frontend holds no secrets and talks to nothing directly — it posts to webhooks, which is why new features ship as new workflows.

The HTML→ACF converter is the fragile link

Conversion is an LLM chain — stochastic and slow. A deterministic replacement is in progress; the publish path is being moved toward convert-once for reliability and fidelity.

The preview companion plugin is additive & reversible

It renders pending content on the live theme behind a single-use signed token, never touches live posts until promote, and leaves no residue when inactive.

Production work is gated

Writes to the production WordPress, n8n, and database are deliberately gated and verified independently after every change. Staging is the build surface.
