WordPress Preview Fidelity — can the preview show exactly what publishes?

A deep research pass into a single requirement: the WordPress preview must show the content exactly as it will appear once live — what the reviewer previews is the published page, 100%. This documents why that is hard today, the architectural unlock that makes it reachable, the solution, the alternatives weighed, an honest 100%-or-not verdict, and how we would prove it before any rollout.

1 The requirement

One sentence, no wiggle room: the WordPress preview must render the content exactly as it will be visible on the live WordPress page. The view in the preview is the view that goes live. Not "like the editor" — like the published page, byte/pixel-faithful.

This is the right bar for healthcare content: a reviewer approves what they see, so what they see must be what readers get. Anything less means approving one thing and publishing another.

2 Why the preview cannot match the page today

A published LifeStance blog post is not HTML. It is a single WordPress ACF block — acf/blog-section-builder — whose JSON data holds the whole article (each section as text_block_N_title + text_block_N_description, with the inner HTML stored verbatim). The live page is produced fresh by the theme expanding that block JSON through blog-section-builder.blade.php (486 lines) on every request.

Publishing is therefore a round-trip through a different representation:

stored ACF blocks ──theme renders──▶ HTML (this is what the editor loads & edits) │ editor edits ───┤ ▼ convertHtmlToAcf (LLM) ──▶ ACF block JSON ──▶ post_content ──theme re-renders──▶ LIVE PAGE

The editor holds the rendered output of the blocks (sections, wrapper divs, utility classes). Publishing re-derives the source (blocks) from that HTML with an LLM, then the theme renders those blocks again. So the live page is a fresh re-render of re-derived blocks.

This is why "Option C" (render the editor's raw HTML) could never satisfy the requirement. It renders the input side of the loop; the published page is the output side. They are different by construction — different DOM, different classes, different layout. It was not a bug to patch; it was the wrong representation.

Two things compound it

  • The converter is a stochastic LLM. convertHtmlToAcf is a ~37-node, ~25-LLM-call chain. The same input produced section counts of 8 / 14 / 15 / 16 across runs, and hard-fails roughly half the time. The published page reflects one particular roll of that dice.
  • Write-time transforms. The publisher applies ~12 per-field byte transforms (unslash, entity-decode, newline→space, quote-escape) — which raw editor HTML has never been through.

3 The unlock — and it is not what we assumed

We assumed the fix was a deterministic converter. The research shows that is not required for fidelity. The unlock is far simpler and bulletproof:

Convert ONCE. Convert the editor HTML → ACF blocks exactly once. The reviewer previews that exact block payload, rendered by the real theme. Go Live publishes that same frozen payload, verbatim — never re-converting.

Because preview and publish then read the identical bytes through the identical theme path, they are identical by construction — even though the converter is random. The stochasticity stops mattering the instant you freeze one roll and ship it to both eyes. You do not need the converter to be deterministic; you need the same single output to feed both preview and publish.

The trap to avoid: conflating "fidelity" with "a perfect converter." They are separate. Convert-once guarantees fidelity. A deterministic converter is about output correctness and speed — a real, separate goal (and the research confirms it is feasible, because the target is one block type whose inner HTML is an opaque passthrough, not an open-ended tag-to-block mapping).

4 The solution — render the real blocks, convert once, promote verbatim

1 · Convert once, at preview time

Editor HTML → ACF block payload, in a bounded retry loop (covers the ~50% hard-fail). Stage that exact payload.

2 · Preview the real render

Reviewer sees the staged payload rendered through the real theme block path (do_blocks → blog-section-builder.blade.php) — not raw HTML.

3 · Go Live promotes verbatim

Go Live publishes that exact payload — no second conversion.

4 · Publisher uses the approved payload

The live publish path consumes the approved blocks rather than re-converting — closing the last gap.

The plugin already implements the preview-render and the verbatim-promote halves. The remaining integration is wiring the live publisher to consume the approved payload, plus the theme-render dependency below.

5 Alternatives weighed

A
Unify on ACF blocks — convert once, promote verbatim, real block render. RECOMMENDED
The only path to true 100% with a correct published page. Half-built already.
B
Raw HTML everywhere (preview and publish both store raw HTML). REJECT — fallback only
Trivially 100%, but it degrades every published page — loses the ACF block layout, CTAs, references, section styling. Parity by making both wrong. Fails the content-quality bar.
C
WordPress native revision/autosave preview. REJECT
Still needs the converter; ACF field context inside revisions is fragile; reviewer has no WP login. No fidelity gain, more moving parts.
D
Shadow / clone post. REJECT
Violates the locked "no clones / no deletes on the client's prod" rule, leaks into sitemaps/search, and still doesn't solve the conversion.
E
Swap-and-capture on the live post. SUBSUMED
The plugin already does the safe, in-memory version of this (swaps only the render output, never the DB). The "swap DB then restore" variant is strictly worse.

6 The honest 100% verdict

True 100% is architecturally achievable and empirically provable — but not with what is deployed today, and it is gated on two requirements, one of which is the client's.

Plain answer to "will it be 100%?" Yes — the mechanism is sound and the hard parts (byte-parity at the assembler, gating, reversibility) are already proven on staging. But the guarantee is contingent on (a) clearing the section-stack 500 on the client's theme so the real block path renders, and (b) wiring publish to use the approved payload. Without both, the only currently-guaranteed-identical option is raw-HTML — which buys parity by degrading the published design, an unacceptable trade for healthcare content.

7 How we would prove it — before any rollout

  1. Convert-once invariant (the load-bearing test). Stage → capture payload P. Publish → read back the published payload P′. Assert P == P′ byte-for-byte. A positive control, not an eyeball. If they differ, no second conversion happened — 100% is structurally impossible and we stop.
  2. Render-equivalence. Render P via the preview path and as a real published page; normalize (strip nonces, cache-busters, timestamps); diff the DOM trees. Zero meaningful node/text/attribute differences = proven.
  3. On the real prod theme (sage11), across flat and structured content, confirming the render traversed the true block path — not the raw-HTML fallback.
  4. Edge/cache parity + gating negatives — anonymous/expired-token requests still see live, never pending.
"100% proven" = all of the above hold simultaneously, on the real theme, across content shapes, via the true block path. That is a demonstrable bar, not a hopeful one.

8 The decision that unblocks everything

The single fork that decides the outcome is the section-stack 500 on the client's theme:

If we can fix / get source access to the prod theme

Approach A delivers true 100%. Next step: the full implementation + proof plan, then a gated staging build, then the one-time prod verification.

If the client's theme is off-limits

The only guaranteed-identical option is raw-HTML-everywhere, which degrades the published design. We'd have to decide whether "preview == publish" is worth that trade — a product call, not an engineering one.
