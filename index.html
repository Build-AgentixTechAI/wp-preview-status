A step-by-step account of today's work (2026-06-17): the speed problem we set out to solve, the deterministic converter we built and shipped, and the separate theme-level error that still stops the preview from displaying — with the exact error, root cause, the proposed fix, and the challenges to landing it.

TL;DR One-minute summary

SOLVED

The speed problem

WordPress Preview took 10–30 minutes to build. We replaced the slow AI conversion step with a deterministic parser. The convert now runs in milliseconds. Deployed to staging, additive, with a proven one-click rollback.

STILL BLOCKING

The preview won't display

After the speed fix, the preview pane shows a 500 error. The cause is a bug in the LifeStance theme (not the converter, not ACF, not our data) that crashes whenever its blog-section-builder block is rendered live. Our preview is the only thing that renders it live.

Bottom line: The thing we were asked to fix (speed) is done. A separate, pre-existing theme bug — which has nothing to do with the converter — is what now stops the preview from rendering. The fix for that lives in our preview plugin and avoids touching the client's theme.

1 The problem we set out to solve

When a reviewer clicks “WordPress Preview” in SmartCMS, the goal is to show the edited blog rendered exactly as it will look on the live LifeStance theme — before publishing. The complaint this morning: it takes 10–30 minutes to load (often spinning at “Building your preview…” for 5+ minutes), which makes the feature unusable.

Target: bring it down to seconds (the brief said “a few minutes, max two”).

2 Deep dive — diagnosing the 30 minutes (step by step)

2.1
Traced the preview pipeline end-to-end. Frontend stages the content, then polls every 12s until the backend returns a preview URL. The frontend is fast (adds ≤12s) — the time is spent server-side, in the HTML→ACF conversion step (convertHtmlToAcf) inside the n8n staging workflow.
2.2
Opened the convert workflow. It is a 37-node, ~25-LLM-node multi-agent AI pipeline — an Orchestrator Agent driving 5 sub-agent tools (Style Analyzer, HTML Parser, Block-Change Detector, Schema Validator, JSON Builder), a RAG vector store, multiple OpenAI models, and 14 output parsers. This entire chain runs on every preview.
2.3
Measured it from real execution history. Each convert attempt takes 1.4–9.6 minutes (avg ~4 min) and fails ~60% of the time. The workflow retries the whole chain up to 3×. 60% failure × multi-minute attempts × 3 retries = the observed 10–30 minutes.
2.4
Proved the AI step is non-deterministic. For the identical input (same 16-heading post), across runs the converter produced section counts of 8, 14, 15, and 16 — it disagrees with itself. This confirmed the slowness is inherent to using a heavyweight LLM pipeline for what is fundamentally a mechanical transform (heading → section, paragraph → block).

3 The solution we implemented — a deterministic converter

Converting HTML to the ACF block structure is a structural mapping, not a reasoning task. So we replaced the AI pipeline (on the preview path) with a small, deterministic parser.

  • Built a dependency-free parser (pure JavaScript, runs natively in an n8n Code node) that maps the editor HTML into the exact ACF field structure. Handles both content shapes (flat editor HTML and pre-structured HTML).
  • Verified it against 22 real conversions — byte-identical output to the “good” AI runs, produced consistently every time.
  • Deployed additively into the staging stage-bridge workflow: the parser runs first; if it ever can't handle an input, it falls back to the old AI converter (so worst case = today's behavior, never worse).
  • Proven rollback — the live workflow was backed up; reverting is a single re-import, demonstrated live (cycled the workflow 16→18→16→18 nodes to prove it).
Result: the convert step went from ~4 minutes (and 60% failure) to under 1 millisecond, 100% consistent. Confirmed live: the preview's backend now fires the deterministic parser and skips the AI entirely. The 30-minute problem is solved.
Frontend → stage request → Deterministic parser (ms) → ACF data → WordPress preview plugin → render

4 The blocker — the preview still won't display

With speed solved, clicking Preview now resolves in seconds — but the preview pane shows an error instead of the page. We chased it to ground in two layers:

4.1 — First a 404 (a data issue, since resolved as a known)

Some previews 404'd simply because the staging WordPress only has 9 posts; many SmartCMS requests point at production post IDs that don't exist on staging. That's a staging-data gap, not a code bug — for a post that does exist on staging, the routing works.

4.2 — Then a 500 Internal Server Error on render (the real blocker)

For a post that does exist on staging (e.g. 137514), the preview gets past the 404 but the WordPress render returns a 500. We proved it's not our content by staging the old AI converter's own known-good output against multiple posts — it 500s identically. So the block content is irrelevant; the render path itself is broken.

5 Root cause & ownership — who's actually at fault

The error is a Blade template (theme) bug. The LifeStance theme is a Sage/Roots theme that renders pages with Laravel “Blade” views. When its blog-section-builder block is rendered live (via WordPress do_blocks → the theme's Blade view), that nested render corrupts the shared Blade “section” counter — so the page wrapper's @endsection then crashes with “Cannot end a section without first starting one.”

Why it only affects the preview: Published posts store the already-rendered flat HTML in the database (we confirmed: their post_content is plain <section class="blog-section-builder">… HTML, not ACF block markup). So the live site never uses the broken block-render path. Our preview is the only thing that renders the block live — so it's the only thing that trips the theme's bug. The bug has effectively always been there; nothing had exercised it before.

6 The proposed fix (and why it's the right one)

We do not need to edit the client's theme. The fix lives in our preview plugin:

Make the preview render the staged content as flat HTML — the exact format the site already stores in post_content — instead of ACF block markup. Flat HTML renders as plain content (no do_blocks, no Blade block view), so the theme's broken path is never entered.
  • Faithful: it's the identical format published posts use, so the preview looks exactly like the published page (we have a real post's stored HTML as the byte-level reference to match).
  • Contained: 100% inside our plugin; the client's theme is untouched.
  • Reuses today's work: the deterministic parser already produces the exact fields needed to rebuild that flat HTML — so the result is a preview that is both fast and actually displays.

7 Challenges to landing the fix (honest)

1 · Live-deployment risk

There's no separate test environment — the plugin change goes directly onto the live staging WordPress. A single PHP syntax slip takes the whole site down (the plugin loads on every request). Mitigated by: backup → syntax-check → verify a normal page still loads → test preview → instant revert.

2 · Fragile server access path

The server is reached through a multi-hop SSH chain into a Docker container. Writing a full PHP file through it reliably needs careful handling (base64 transfer) to avoid quoting errors.

3 · Format fidelity across all shapes

We have one solid reference post to match. Reconstructing the exact published HTML for every content shape — images, the blue CTA, reference lists, banner/intro, multi-section — may need iteration and a few more reference posts to be byte-faithful everywhere.

4 · It's a write to client infrastructure

Standing rule is read-only on that server by default; deploying the fix is a mutation on a client system and needs explicit sign-off — especially carefully, after today.

Not a challenge: the logic is well-understood (we have the parser and the exact target format), and the render code only affects previews — so logic errors can't break published pages (only syntax errors can, which the safety drill guards against).
