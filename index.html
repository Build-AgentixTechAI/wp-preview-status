2 The problem — why preview took up to 30 minutes (and broke)

When a reviewer opened WordPress Preview, SmartCMS rendered the draft on the real LifeStance theme so the reviewer saw exactly what readers would see. To do that, the editor's content had to be turned into WordPress "ACF blocks" and then rendered. Two independent things made that slow and fragile:

A
The conversion was an AI pipeline — slow and unpredictable. Turning the editor HTML into ACF blocks ran through a 37-node, ~25-step LLM chain (convertHtmlToAcf). A single fresh conversion could take many minutes, and it was stochastic: roughly half of runs hard-failed and had to be retried, and the output shape varied run to run. Retries stacked on retries — that is where "up to ~30 minutes" came from.
B
Even when it converted, the render itself crashed with a 500. The LifeStance theme (a Sage/Blade theme) has a defect in its section-builder that throws a fatal error when a block is rendered live. This is in the client's theme, not in SmartCMS — the AI pipeline's own output triggered the same 500. So even a "successful" conversion frequently produced a blank, errored preview.
Old preview path: editor HTML → convertHtmlToAcf (LLM ×25, minutes, ~50% fail) → ACF blocks → theme render → 500 (theme bug)
Net effect for the user: a spinner for many minutes, often ending in an error rather than a preview. Unusable for real review work.

3 The solution we shipped

Two changes, layered. The second is the one that fundamentally fixes it.

3.1  A deterministic converter (replaces the slow AI step)

We built a deterministic HTML→ACF converter — plain, rule-based parsing, no AI, no external dependencies. It produces the same output every time and runs in milliseconds instead of minutes. It was deployed additively to the staging bridge: the deterministic path runs first, and the old LLM pipeline remains as an automatic fallback for the rare shapes it does not yet cover. Fully reversible.

3.2  Render the published format directly — "Option C" (the key fix)

The decisive insight: a published LifeStance post already stores the editor's HTML as its content — byte-for-byte. So there is no need to reconstruct ACF blocks and push them through the buggy block-render path at all for a preview. Instead, the preview renders the editor HTML directly, which is exactly the format a published post keeps.

New preview path: editor HTML → render directly as post content → live theme → 200, in seconds

Because the preview uses the same bytes that a publish writes, two problems disappear at once: the slow/stochastic conversion is no longer on the preview path, and the theme's block-render bug is never triggered (we never enter the broken block path).

Result. Preview renders in seconds, returns 200 reliably, and is byte-identical to what will publish — so "what you preview is what goes live" is true by construction, not by hope.

4 Why this is the right solution for this situation

It was chosen against the realistic alternatives, on the constraints that actually matter for healthcare content.

✓ Absolute fidelity

The reviewer sees the exact bytes that publish, on the real theme. No preview-vs-live drift — critical when wrong/var­ied content on a clinical page is the worst outcome.

✓ Fast

No AI conversion on the hot path. Milliseconds of work + a normal page render = seconds, not minutes.

✓ Reliable

Removes the ~50%-failure LLM step and side-steps the client theme's render defect entirely. Deterministic in, deterministic out.

✓ Lean & reversible

Reuses a format WordPress already stores. No new plugin, no rewrite of the publish data model, additive deploy, proven rollback, staging-isolated.

Why not the alternatives

5 Go Live — what we built

Once a reviewer can faithfully preview the content, the natural next action is "publish exactly this." That is the new Go Live button inside the preview screen.

One click takes the content you are previewing and pushes it straight to the live published post, then closes out the SmartCMS request so it no longer sits as an open item. It is a single, atomic action:

Go Live: click → goLivePreview → promote previewed content to the live post (Option C) → resolve the request → return to the queue

"Resolve the request" means, in one transaction: the request status becomes Published, its approval workflow is closed, and an audit note Published via Go Live is recorded — so the trail honestly shows this content went live through the fast path rather than the step-by-step approval chain. Because there is no live auto-refresh in the app, the screen then returns to the approvals queue, which reloads and shows the request as resolved.

By design, Go Live has no additional approval gate. Anyone who can open the preview and click Go Live publishes immediately. That is the intended behaviour for a fast, authority-holder publish — and it is exactly the dimension where it differs from Approve And Post (next section).

6 Go Live vs. Approve And Post — the difference

Both buttons end with content on the live site, but they represent two different intents. One moves a request through governance; the other publishes immediately.

GOVERNED Approve And Post

Advances the request along its approval workflow. The approver reviews the draft and approves the current step. If more approval steps remain, the request routes to the next approver; only when the final step is approved does the content publish through the standard publish pipeline. Every step is recorded.

Use it when: the content should pass through the normal multi-role editorial review before it goes live.

DIRECT Go Live

Publishes the previewed content immediately, bypassing any remaining approval steps, and marks the request Published. No routing to a next approver — it goes live on click, using the exact bytes shown in the preview (Option C).

Use it when: an authorised person has previewed the content and wants it live now, without further approval hops.

In one line: Approve And Post = "send this forward through the approval chain (and publish when the chain completes)." Go Live = "publish exactly what I'm previewing, right now." They are complementary, not duplicates — a governed lane and an express lane.

7 Safety, status & what's next

  • Staging only. All of the above is live on staging. Production is untouched and remains gated behind explicit sign-off.
  • Reversible. The converter is additive with the old pipeline as fallback; the Go Live backend (webhook + database resolution) is backed up and can be rolled back; the WordPress plugin changes have saved backups.
  • Verified by outcome, not by status code. Go Live was proven end-to-end on staging: the live post updates to the previewed content and the request is confirmed resolved in the database — not merely a 200 response.
  • Open follow-ups. (1) The LifeStance theme's block-render defect is a separate upstream item to report. (2) A small share of richly-structured drafts still fall back to the AI converter — a future iteration. (3) Production rollout of both the preview fix and Go Live awaits sign-off.
Bottom line. Preview is fast and faithful, Go Live gives a clean one-click publish alongside the governed approval flow, and nothing touched production. Ready for review and a production go/no-go.
