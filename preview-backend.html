1 The moving parts

The preview is produced by four cooperating pieces. The frontend never talks to WordPress directly — n8n is the boundary, and a companion plugin on the WordPress side does the reviewer-only render.

3 Step 1 — fetching the live blog

The preview must render against the real published post on the real theme, so the first job is to resolve which WordPress post we're previewing.

a
The frontend sends a publish reference, not a post id. SmartCMS knows the editorial object (publishId = the publication_requests row), plus the reviewer's current editor HTML (fullHtmlContent).
b
The bridge resolves it to the WordPress post. A Supabase read on publication_requests maps publishId → source_blog_id — the actual WordPress post ID on the live site. It also pulls the AI session for context. So "fetching the live blog" = resolving the editorial request to its live WordPress post, by id.
Why by id, not by re-fetching content: we don't copy the live blog's HTML to render it. We render on the live post (same id, same theme, same templates), and only substitute the body at render time. That's what keeps the preview faithful to the live site's chrome, CSS, and layout.

4 Step 2 — staging the proposed content

The bridge sends the reviewer's edited HTML to the plugin's /stage endpoint, which stores it as a pending layer on the post — never touching the live content.

stage bridge ──▶ POST /wp-json/smartcms-preview/v1/stage/{post_id} body: { "html": <the editor's content> } plugin /stage │ store pending HTML in post meta _smartcms_pending_html │ mint a signed, post-scoped, single-use token ▼ return { staged, token, preview_url }
  • Pending, not live. The proposed content lives in an isolated post-meta key. The live post_content is untouched until Go Live, so the public never sees pending content.
  • Option C — the editor HTML directly. What's staged is the editor's own HTML (the same bytes the reviewer is editing), not a reconstructed format. This is the key render decision (see §10).
  • Capability-gated. /stage requires a logged-in user with edit_posts — the bridge calls it with the SmartCMS service credential.

5 Step 3 — rendering the exact view (the heart of it)

The reviewer's iframe loads the live post's own URL with the token. The plugin intercepts the render and swaps the body for the pending content — so the proposed edits are painted by the real theme.

a
Token → cookie exchange (template_redirect). The first request carrying a valid token burns it (single-use), sets a short-lived httpOnly cookie, and redirects to a clean URL (?smartcms_pv=1) so the token never lingers in history or referrers.
b
Authorization check. The render classifies the request: ok (cookie authorizes the staged render), blocked (a preview was attempted but isn't authorized → an explicit "preview unavailable" notice), or none (not a preview request → the live page, untouched).
c
The body swap (the_content filter, priority 8). For an authorized request on the queried post, the plugin returns the staged pending HTML instead of the live post_content. WordPress and the theme then render that exactly as they render any post body — same wrappers, same CSS, same templates.
The result: the reviewer sees their proposed content laid out by the real LifeStance theme, at the real post's URL — but nobody else does, and the live post is unchanged.

Cache & isolation

  • Preview responses set no-store + Referrer-Policy: no-referrer — never cached, token never leaks via Referer.
  • The swap only touches the main queried post — related/recent-post widgets that also run the_content stay live.

6 Auth & security model

The reviewer has no WordPress login. A signed token is the authorization — layered to keep it safe.

ControlHow
Token integrityHMAC-signed payload post | user | exp | nonce — can't be forged or retargeted to another post.
Short-lived15-minute TTL; the frontend mints a fresh token on every preview open / Refresh.
Single-useThe token's nonce is burned on first render; a replayed token shows the "unavailable" notice, never content.

AuditMint / render / promote events are logged to a rolling audit on the post.

7 Draft posts

Editors preview drafts too — but a draft has no public front-end, so WordPress would 404 it before the render swap could run. The plugin handles this with a pre_get_posts hook: when a valid preview token/cookie is present, it pins the main query to exactly the authorized post id and allows non-public statuses — so the draft resolves and the normal render path runs. Without a valid token, drafts 404 as usual (no exposure).

8 The references repair

A real-world wrinkle: the collaborative editor (TipTap) mangles the original numbered references list — it strips the styling class and flattens the nested <ol><li><ul> into fragmented per-item <ul> lists. The live theme globally resets list markers, so those fragments render without numbers.

So the render path (and the Go Live promote) run a scoped repair: using a real HTML parser (PHP DOMDocument), it rebuilds only the references block into one clean <ol class="list-decimal"> — the exact structure the theme numbers. It is fail-open (any parse issue returns the original) and never touches the article body, so it can't affect layout.

Applied in two places, identically: at render (so the preview shows numbered references) and at promote (so Go Live writes the same clean list to the published page). Preview == published for references.

9 Go Live — promote to the live post

When the reviewer is happy, one click publishes exactly what they previewed.

Go Live ──▶ goLivePreview webhook │ resolve publishId → source_blog_id │ POST plugin /promote/{post_id} │ write the staged HTML (references-repaired) into live post_content │ resolve the request in Supabase: status → Published, workflow closed, audit logged ▼ the live page now shows exactly the previewed content

Promote writes the same staged bytes the reviewer saw — so what you preview is what publishes. The pending layer is then cleared.

10 Why this design — and the honest limit

Companion plugin, not a replica

Rendering on the real WordPress + real theme is the only way to get the live site's true CSS, templates, and chrome. A replica renderer could always drift.

Additive & reversible

The plugin only adds a pending meta layer + a gated render filter. It never mutates a live post until Go Live, and leaves no residue when inactive.

Option C — render the editor HTML directly

Fast (no slow AI conversion on the path) and it skips a fatal bug in the theme's block renderer. Faithful for flat content.

Token, not a WP session

Reviewers have no WordPress accounts; a signed, single-use, short-lived token authorizes exactly one post's preview and nothing else.

The honest limit. A published LifeStance post is stored as structured ACF blocks that the theme re-renders; the editor edits the rendered HTML. Option C previews that HTML directly — perfectly faithful for simple content, but for richly-structured posts it isn't pixel-identical to the block-rendered published page. The path to true 100% (render the actual blocks via the theme, converting once and publishing that exact payload) is documented separately; it depends on clearing a section-render fault in the client's theme.
