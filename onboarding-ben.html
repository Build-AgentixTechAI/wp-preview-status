1 What SmartCMS is

An editorial workflow layer on top of the client's WordPress. Teams draft and collaboratively edit blog content, route it through multi-role approval, preview it on the live WordPress theme, and publish AI-assisted content.

The stack, in one line: React frontend (Lovable) → n8n webhooks (the only integration boundary) → Supabase (editorial data) · WordPress (publishing) · Hocuspocus (collab editing) · AI (HTML→ACF conversion).

Key fact to internalize: the frontend talks to nothing directly — every action is an n8n workflow. And a published post is stored as ACF blocks the theme re-renders, while the editor edits the rendered HTML. That gap drives most of the hard problems (see the preview docs in §8).

4 Run it locally

npm install
npm run dev        # Vite dev server on http://localhost:8080
npx vite build     # build (use this to type-check your changes)

Your local frontend at localhost:8080 hits the staging backend (staging n8n + staging Supabase + staging WordPress) — so you can test full flows locally without deploying. You'll need a SmartCMS login (ask Rubeena for staging credentials).

There's a .env with the staging config (base URL, Supabase). It comes from the onboarding bundle / Rubeena — it is not committed.

7 The freeze & working rules

Active deploy freeze: no deploys to the staging WordPress plugin or staging n8n until Rubeena explicitly lifts it. Pushing to git is fine; deploying to the live staging surfaces is not.
  • Coordinate with Lev — both of you can push. Branch your work; don't push main without a heads-up.
  • Verify outcomes, not status codes. An HTTP 200 ≠ correct — read the DB / live state and diff before/after.
  • Back up before any live mutation, and keep a tested rollback. Reads are free; writes are gated.
  • Production is off-limits without explicit, per-change approval.
  • Don't trust memory for "what's deployed" — check the live artifact (the running plugin, the n8n execution, the served bundle).
